# 开发上下文与技术债清单

## 1. 当前开发上下文（对后续迭代最重要的信息）

- 这是一个**本地桌面应用**，非 SaaS，多用户协作/权限暂不在架构内。
- 后端不是 HTTP 服务，而是 Tauri Command RPC。
- 数据库是本地 SQLite 文件，生命周期跟客户端机器绑定。
- 项目以“功能可用”为主，工程化层面（测试、分层、约束）尚有较大提升空间。

## 2. 关键隐含规则

- 学生、商品、购买记录都属于班级上下文。
- 购买记录里 `points` 存的是订单总积分（`单价 * 数量`）。
- 发货状态当前核心值是：`pending`、`shipped`（兼容历史 `delivered` 显示）。
- 学生积分可在前端快捷增减，当前未做后端下限保护。

## 3. 已识别问题与风险

## 3.1 高优先级

- 学生学号更新链路不完整：
  - 后端 `UpdateStudentRequest` 支持 `student_number`。
  - 前端 `studentApi.update` 当前未透传该字段。
  - 结果：UI 可编辑学号，但更新可能未落库。

- 数据一致性约束偏弱：
  - 无唯一约束（如同班学号唯一）。
  - 无检查约束（积分/库存可为负值）。
  - 外键策略未在运行时统一显式化。

- 大文件维护风险：
  - `ClassStudents.tsx`、`ClassProducts.tsx`、`database.rs` 体量大，逻辑集中，回归风险高。

## 3.2 中优先级

- 购买弹窗“购买后剩余积分”显示计算未考虑购买数量（只减了单件积分）。
- 购买记录页面调用了 `useToast` 但未渲染 `ToastContainer`，提示可能不可见。
- 导入流程逐条串行调用 command，数据量大时耗时明显。
- `StudentModal.tsx` 当前未被页面使用，存在双实现并存的认知负担。

## 3.3 低优先级

- 样式逻辑与业务逻辑混杂，后续主题或皮肤切换成本高。
- 缺少统一埋点/日志上下文，排查线上（用户端）问题成本偏高。

## 4. 测试与质量现状

- 当前仓库无自动化测试（单测/集成/E2E）。
- 未见 schema migration 测试与关键事务回归测试。
- Excel 导入边界（空值、重复值、异常格式）主要靠运行时 try/catch。

## 5. 建议的迭代优先级（建议先做）

1. 修复学生学号更新透传问题（小改动高收益）。
2. 补充关键数据约束（最少唯一约束 + 非负约束）。
3. 修复兑换弹窗剩余积分显示逻辑。
4. 为购买记录页补齐 ToastContainer。
5. 把 `ClassStudents` 按 hooks + 子组件拆分。
6. 为购买事务、积分更新、Excel 导入补最小测试集。

## 6. 推荐的文档与规范补充

- `docs/业务规则字典.md`（积分、发货状态、导入模板字段标准）。
- `docs/错误码规范.md`（从字符串错误升级到可枚举错误码）。
- `docs/迁移策略.md`（数据库 schema 版本化与回滚策略）。
- `docs/测试策略.md`（前后端最小可执行测试清单）。

## 7. 后续协作建议

- 先定“数据一致性最小基线”，再继续加功能。
- 每次新增页面前先定义接口契约，再写 UI，避免前后端语义偏移。
- 将“批量导入导出”沉淀为独立服务模块，减少页面重复代码。
